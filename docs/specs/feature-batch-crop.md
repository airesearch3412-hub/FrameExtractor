# 功能規格：批次裁剪（✂ 批次裁剪）

> 範例規格，示範 GWT 寫法。對應 [DEC-004](../../decision_log.md)。

---

## 一、概述

| 欄位 | 內容 |
|---|---|
| 功能名稱 | 批次裁剪 |
| 對應分頁 / CLI | GUI「✂ 批次裁剪」分頁 / `crop_images.py` |
| 優先級 | P1 |
| 相關決策 | DEC-004（本地實作）、DEC-006（預覽縮放/平移） |

## 二、使用者故事

> 作為使用者，我想對一批**相同尺寸**的截圖在預覽上**框選同一裁剪區域**並輸出成**統一格式**，以便快速去除電子書頁面的外框雜訊。

## 三、驗收標準（Given / When / Then）

| 編號 | Given | When | Then | 類型 |
|---|---|---|---|---|
| GWT-1 | 清單有 N 張相同尺寸圖片、已框選裁剪範圍、已選輸出資料夾 | 按「開始裁剪」 | 每張依同一裁剪框輸出到該資料夾（格式統一、可選統一尺寸），並產出 `_crop_report.csv` 與 `_crop_summary.txt` | 主流程 |
| GWT-2 | 某張圖片無法讀取 | 處理到該張 | 標記 `read_failed` 並繼續處理其餘，不中斷整批 | 錯誤處理 |
| GWT-3 | 輸出檔會覆蓋來源原檔，或與另一來源同名 | 寫出該張 | 同源者略過（`skipped_overwrite_source`）、跨來源同名者自動改名 `__2`，避免資料遺失 | 邊界 |
| GWT-4 | 某張尺寸與首張不同 | 處理該張 | 記錄尺寸不一致警告但仍套用同一裁剪框；裁剪框超界時夾住並記 `clamped` | 邊界 |

## 四、邊界情境

1. **裁剪框反向 / 無效**（右≤左 或 下≤上）：GUI 會把數值夾正、CLI `--crop` 直接報錯，不得開始裁剪。
2. **使用者中止**：回報 `stopped=True`、顯示「已中止」，不得謊報「完成」；已處理數如實計入。

## 五、測試映射

| 驗收 | 驗證方式 | 證據 / 結果 |
|---|---|---|
| GWT-1 | CLI 端到端 + headless `BatchCropWorker` | 4 張輸出皆 1320×919、report/summary 產出 ✅ |
| GWT-2 | headless（含損毀/不可讀檔） | 標記 read_failed、整批續跑 ✅ |
| GWT-3 | headless（`A/page.jpg`+`B/page.jpg`、輸出=來源） | 產出 `page.jpg`+`page__2.jpg`；同源略過 ✅ |
| GWT-4 | CLI（混入 1280×720） | 警告 + 夾住輸出 ✅ |

## 六、非目標（Out of Scope）

- 不做不同尺寸圖片的「各自智慧裁剪」（本功能假設同尺寸、同一裁剪框）。
- 不做裁剪後去重（屬另一待確認需求）。
